Study Guide: LLM Inference + Observability

What this assignment is

You are building a text-to-SQL system backed by a self-hosted LLM, then operating it: measuring it, iterating on it, and proving it meets a latency/throughput SLO without breaking quality. The scenario is an internal analytics product where analysts type questions in English and the system returns rows from a data warehouse.

The stack has two layers of observability that answer different questions:

Concepts and technology

vLLM and LLM inference

A language model produces one token at a time. Serving it efficiently requires batching many concurrent requests — tokens for different sequences are computed in the same GPU kernel. vLLM does this through two mechanisms.

PagedAttention

The KV cache (the key/value tensors representing a sequence's history) is stored in fixed-size pages of GPU memory, like OS virtual memory. This eliminates external fragmentation: sequences of different lengths share the same physical memory pool, and memory is only allocated as tokens are generated.

Continuous batching

Rather than waiting for all sequences in a batch to finish before starting new ones, vLLM inserts new requests as soon as a slot frees. This keeps GPU utilization near-constant under load.

The prefill / decode split

Every request has two phases. Prefill: the entire prompt is processed in one forward pass — this is compute-bound and fast. Decode: tokens are generated one at a time — this is memory-bandwidth-bound and slow.

Key vLLM flags for this workload

Workload profile: 1.5–3K-token prompts, short structured outputs (SQL), 2–3 serial LLM calls per user request, on H100 80GB.

FlagRationale
--max-model-len 8192Qwen3 defaults to 32K+ context. Reducing it cuts KV cache memory proportionally, fitting more concurrent sequences. Workload never needs more than ~4K tokens.
--enable-chunked-prefillSplits long prefill operations across steps, interleaving with decode. Reduces P99 TTFT spikes under load.
--gpu-memory-utilization 0.95H100 80GB is reliable. Pushing above the 0.9 default recovers ~4GB for the KV cache pool.
--max-num-seqs 128Upper bound on concurrent sequences. Start here; back off if you see OOM or scheduler thrash.
--enable-reasoning --reasoning-parser deepseek_r1Qwen3 emits <think>…</think> blocks. Without this they appear in content. With it, routed to reasoning_content and stripped.

Qwen3-30B-A3B: Mixture-of-Experts

Qwen3-30B-A3B is a Mixture of Experts (MoE) model. Each Transformer layer replaces its dense feed-forward network with N independent "expert" networks. A learned router selects k experts per token. Only those k experts activate; the rest sit idle.

Implication for vLLM: the KV cache is sized per active layer, not per total parameter. MoE models are often more memory-efficient to serve than their parameter count suggests. On a single H100 80GB, Qwen3-30B-A3B loads comfortably.

BIRD-bench and execution accuracy

BIRD-bench is an academic benchmark of natural-language questions paired with gold SQL over real databases. The eval metric is execution accuracy: run both the gold SQL and the model's SQL, compare result sets. Two different SQL strings that produce the same rows count as equivalent.

LangGraph: stateful agent graphs

LangGraph models an agent as a directed graph where nodes are Python functions and edges are fixed or conditional. State — a dataclass — is threaded through every node. Each node receives the current state and returns only the fields it changed.

The verify → revise loop is a self-consistency-inspired refinement: generate a candidate, evaluate it, and if it fails, fix it given the failure mode. The iteration cap (MAX_ITERATIONS = 3) is critical for SLO compliance.

Prometheus and Grafana

Prometheus scrapes /metrics from vLLM every 5 seconds. vLLM exposes histograms, counters, and gauges. The key metrics by category:

CategoryMetricPromQL typeWhat it tells you
Latency vllm:e2e_request_latency_seconds_bucket histogramFull round-trip, first token to last
vllm:time_to_first_token_seconds_bucket histogramPrefill cost — high = prompt too long
vllm:time_per_output_token_seconds_bucket histogramPer-token decode — high = memory BW bound
Throughput vllm:generation_tokens_total counter → rateTokens out per second
vllm:request_success_total counter → rateRequests served per second
vllm:num_requests_running gaugeSequences actively decoding now
KV Cache vllm:num_requests_waiting gauge>0 means KV cache is full, requests queuing
vllm:gpu_cache_usage_perc gaugeFraction of KV pages in use (0–1). >0.9 = danger zone

PromQL for a histogram P95: histogram_quantile(0.95, sum(rate(vllm:e2e_request_latency_seconds_bucket[1m])) by (le))

Langfuse: LLM observability

Langfuse captures traces: one trace per agent run, with nested spans for each LangGraph node. Each span records the prompt, response, latency, and token counts. The langfuse.langchain.CallbackHandler integration instruments LangGraph automatically.

Phase 3 Agent implementation

The graph wiring is already in agent/graph.py. generate_sql_node is a worked example. You implement three things.

1. Prompts in agent/prompts.py

  • GENERATE_SQL_SYSTEM: SQL expert role; output only SQL in a code fence.
  • GENERATE_SQL_USER ({schema}, {question}): schema then question; use only listed tables/columns.
  • VERIFY_SYSTEM / VERIFY_USER: judge whether {result} plausibly answers {question}. Return {"ok": bool, "issue": str}. Catch: SQL errored, zero rows when rows expected, columns irrelevant to question.
  • REVISE_SYSTEM / REVISE_USER: given failing {sql}, {result}, and {issue} — produce corrected SQL.

2. verify_node

def verify_node(state: AgentState) -> dict:
    response = llm().invoke([
        ("system", prompts.VERIFY_SYSTEM),
        ("user", prompts.VERIFY_USER.format(
            question=state.question,
            sql=state.sql,
            result=state.execution.render(),
        )),
    ])
    import json, re
    m = re.search(r'\{.*\}', response.content, re.DOTALL)
    try:
        parsed = json.loads(m.group()) if m else {}
    except json.JSONDecodeError:
        parsed = {}
    return {
        "verify_ok": bool(parsed.get("ok", False)),
        "verify_issue": str(parsed.get("issue", "unspecified")),
    }python

3. revise_node and route_after_verify

def revise_node(state: AgentState) -> dict:
    response = llm().invoke([
        ("system", prompts.REVISE_SYSTEM),
        ("user", prompts.REVISE_USER.format(
            schema=state.schema, question=state.question,
            sql=state.sql, result=state.execution.render(),
            issue=state.verify_issue,
        )),
    ])
    sql = _extract_sql(response.content)
    return {
        "sql": sql,
        "iteration": state.iteration + 1,
        "history": state.history + [{"node": "revise", "sql": sql}],
    }

def route_after_verify(state: AgentState) -> str:
    if state.verify_ok or state.iteration >= MAX_ITERATIONS:
        return "end"
    return "revise"python

Phase 5 Offline evaluation

Implement the two stubs in evals/run_eval.py (eval_one and summarize). Helpers run_sql, canonicalize, matches are provided.

def eval_one(question: dict, agent_url: str) -> dict:
    import httpx
    data = httpx.post(agent_url,
                       json={"question": question["question"], "db": question["db_id"]},
                       timeout=120).raise_for_status().json()

    gold_ok, gold_rows, _ = run_sql(question["db_id"], question["gold_sql"])

    # Score each iteration using the SQL from history
    per_iter = {}
    for i, entry in enumerate(data.get("history", [])):
        ok, rows, _ = run_sql(question["db_id"], entry.get("sql", ""))
        per_iter[i] = matches(gold_rows, rows) if gold_ok else False

    final_ok, final_rows, _ = run_sql(question["db_id"], data.get("sql", ""))
    return {
        "db_id": question["db_id"],
        "agent_sql": data.get("sql"),
        "gold_sql": question["gold_sql"],
        "iterations": data.get("iterations", 0),
        "final_correct": matches(gold_rows, final_rows) if gold_ok else False,
        "per_iter_correct": per_iter,
    }


def summarize(results: list[dict]) -> dict:
    n = len(results)
    max_iter = max((r["iterations"] for r in results), default=0)
    per_iter_rates = {}
    for k in range(max_iter + 1):
        correct = 0
        for r in results:
            last = max((i for i in r["per_iter_correct"]), default=-1)
            if k <= last:
                correct += int(r["per_iter_correct"].get(k, False))
            else:
                correct += int(r["final_correct"])  # carry-forward
        per_iter_rates[str(k)] = correct / n if n else 0.0
    return {
        "n": n,
        "overall_pass_rate": sum(r["final_correct"] for r in results) / n if n else 0.0,
        "per_iteration_pass_rate": per_iter_rates,
    }python
